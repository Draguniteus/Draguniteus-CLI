## The Problem We Solve

MiniMax's M2.7/M2.5/M2.1/M2 model family has a **200k context window** and genuine **agentic capability**. But every general-purpose coding tool — Cursor, Copilot, Claude Code — was built to route between many backends. That means:

- The `ThinkingDelta` streaming protocol is an afterthought, not the backbone
- Multi-model parallel orchestration is a hack, not a design principle
- Self-correction is tacked-on, not a first-class loop
- Model-specific optimizations get lowest-common-denominator treatment

**Draguniteus was built in reverse.** We started with MiniMax's native capabilities and built the agent around them.

---

## What This Means for MiniMax Models

When you run Draguniteus with MiniMax-M2.7, you are running the model in the **most optimized environment we know how to build** for it:

| Feature | How Draguniteus Uses It |
|---------|------------------------|
| `ThinkingDelta` streaming | Flicker-free TUI renders thinking vs text as separate delta streams in real-time |
| `interleaved-thinking` beta | Thinking Router auto-injects it for analysis/debug/plan tasks, bypasses for write/create/fix |
| 200k context window | Conversation archive compresses and preserves full context across unlimited turns |
| M2.7/M2.5/M2.1/M2 lineup | Arena Mode runs all four simultaneously, winner-take-all quality comparison |
| MiniMax API tool calling | Native streaming tool execution with 500KB result caps and 10MB MCP line limits |

## How We Built It: Architecture Decisions

```
src/draguniteus/
├── agent.py                     # Streaming tool-loop with self-correction
├── thinking_router.py          # Keyword + token-bucket → interleaved-thinking beta
├── role_adapter.py             # Per-turn developer message (2048 token cap)
├── self_correction.py          # Write→Verify→Fix loop (Python/ESLint/Shellcheck)
├── checkpoint.py               # Atomic saves, 20-file retention pruning
├── streaming_display.py         # Rich.Live — one render per agent thread
├── orchestrator.py            # ThreadPoolExecutor — true parallel model runs
│
├── memory/
│   ├── manager.py              # DRAGUNITEUS.md + ChromaDB + daily notes
│   ├── semantic_graph.py       # File relationship + decision tracking
│   ├── pattern_library.py      # Learned tool sequences (singleton)
│   └── vector_store.py         # ChromaDB similarity search
│
├── workflows/
│   └── agentic_workflow.py    # PLANNING→EXECUTING→VERIFYING→ITERATING
│
├── tools/
│   ├── nested_tool_executor.py # 5-level depth, 3x retry with backoff
│   ├── dynamic.py              # Runtime custom tool builder
│   ├── reflection.py           # Per-tool call stats
│   └── ...
│
└── preview/
    └── server.py              # Port 7420, markdown→HTML rendering
```

**Key design decisions:**
- `ThinkingDelta` vs `text_delta` — both captured and displayed separately in TUI
- Agentic workflows are serializable — state survives crashes
- Memory has three tiers — project conventions (static), semantic graph (persistent), ChromaDB (searchable)
- Self-correction uses real language-specific checkers, not just regex

## Why MiniMax Is the Right Model Family for This

Most agents route to GPT-4o or Claude because of brand recognition. But:

- **MiniMax's `interleaved-thinking` beta** gives the model genuine deliberative reasoning that other models lack as a first-class API feature
- **The 200k context window** means Draguniteus can hold an entire medium-sized codebase in memory while orchestrating agents
- **The M2.7/M2.5/M2.1/M2 lineup** provides natural tiering for arena comparison — the model family is designed for exactly this use case
- **MiniMax's streaming protocol** with `ThinkingDelta` is more granular than what most competitors expose

Draguniteus proves this stack works by running all four models simultaneously on every non-trivial task.

## Known Limitations

| Limitation | Impact | Mitigation |
|------------|--------|------------|
| ChromaDB requires `pip install chromadb` | Not zero-dependency | Falls back to file-based memory if not installed |
| Self-correction maxes at 3 iterations | Complex bugs surface to user | Increase via `max_iterations` config |
| Nested tool depth limited to 5 levels | Very deep call stacks truncated | Flatten tool design where possible |
| Message truncation keeps last 50 turns | Older context may be forgotten | Periodic summary injection planned |
| Checkpoint saves add ~150ms overhead | Latency on every 5th turn | Batch writes when possible |
| Windows: PowerShell execution policy | Scripts may be blocked | Use `-Bypass` flag or `Set-ExecutionPolicy` |
| MiniMax API key required | Not usable without API access | Set `MINIMAX_API_KEY` environment variable |
